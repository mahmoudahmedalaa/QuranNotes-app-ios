## 4. Critical Context

### Architectural Decisions
- **Null-safe Firebase pattern**: `auth`, `db`, `app` are exported as `null` when the API key is missing rather than throwing. This means **every consumer of `auth`** should ideally have a null guard. We only added guards in `RemoteAuthRepository.ts`; other files may still assume `auth` is always present. This works because the `.env` file is now present, but be aware of this if the `.env` is ever lost again.

### Tricky Bugs Encountered
- **White screen crash loop**: The missing `.env` meant `firebaseConfig.apiKey` was `undefined`. `firebase.initializeApp()` was called with bad config, then `firebase.auth()` threw `auth/invalid-api-key`. The error modal couldn't be dismissed, locking the user in a white screen. Fixed by guarding init and exporting null fallbacks.
- **Metro cache**: After creating `.env`, the app still crashed until we killed the Metro process and restarted with `--clear` to force re-reading env vars.

### User Preferences
- The user wants the dashboard to feel **more alive and vibrant** — not dull. Color and gradient prominence matter.
- The user prefers fixes that work **on the iOS simulator** — do not attempt to test via localhost/browser. This is a React Native app, not a web app.
- The user can be impatient — be direct, don't ask unnecessary questions, and don't make mistakes that break the app.
- The user has been working on **archiving/TestFlight builds** but paused that to fix the crash. They may want to revisit archiving.
