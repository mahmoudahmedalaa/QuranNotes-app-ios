### Core Files (read these first)

| File | Purpose |
|------|---------|
| `src/features/auth/infrastructure/ProContext.tsx` | **Central subscription state** — `isPro`, `loading`, `checkStatus()`, `restorePurchases()`. 76 lines. |
| `src/features/payments/infrastructure/RevenueCatService.ts` | RevenueCat SDK wrapper — init, offerings, purchase, `isPro()` check. 123 lines. |
| `src/features/payments/presentation/PaywallScreen.tsx` | Purchase UI screen |
| `app/(tabs)/settings.tsx` | Settings — previously had a dev toggle for Pro status |

### How isPro is determined
1. `ProProvider` runs `checkStatus()` on mount and when `user` changes
2. `checkStatus()` calls `revenueCatService.getCustomerInfo()`
3. `revenueCatService.isPro(customerInfo)` checks `customerInfo.entitlements.active['pro_access']`
4. If entitlement exists → `isPro = true`

## 3. Investigation Hypotheses (Use Systematic Debugging)

> IMPORTANT: Use the systematic debugging skill at `.agent/skills/systematic-debugging/SKILL.md`

### Hypothesis 1: `loading` state race condition
- `isPro` starts as `false`, `loading` starts as `true`
- But if features check `isPro` **without also checking `loading`**, they might not gate correctly
- **Verify:** Search all `isPro` consumers — do they also check `loading`?
- Some features like `KhatmaContext.tsx` DO check: `if (proLoading) return false;`
- Others might NOT check loading and default to allowing access

### Hypothesis 2: RevenueCat initialization failure silently grants access
- If `revenueCatService.getCustomerInfo()` throws, `checkStatus()` catches and logs warning
- `isPro` stays at its previous state (which defaults to `false`)
- But check: does any error handling path accidentally set `isPro = true`?

### Hypothesis 3: Entitlement identifier mismatch
- Service checks `customerInfo.entitlements.active['pro_access']`
- Verify in RevenueCat dashboard that the entitlement is named exactly `pro_access`
- If renamed/misconfigured, `isPro()` could return unexpected results

### Hypothesis 4: RevenueCat returning stale/cached entitlements
- `Purchases.getCustomerInfo()` may cache data
- New account might be picking up entitlements from a previous user session
- Check if `Purchases.logIn()` or `Purchases.logOut()` is called on auth state change

### Hypothesis 5: Features not gating on `isPro` at all
- Some features mentioned as Pro gates in KI might not have actual `isPro` checks in their screen code
- Cross-reference which screens the user accessed vs which have actual guards

### Hypothesis 6: Bug introduced in recent code changes
- Check git diff for any accidental changes to ProContext, RevenueCatService, or gating logic
