# Handover — Fix Per-User Data Isolation on Logout

**Date:** 2026-02-28  
**Priority:** Critical — requires a new archive/build after fixing

---

## 1. The Bug

**Data persists across accounts on the same device.** When a user logs out and another user logs in, the new user sees the previous user's khatma progress, completed surahs, mood data, notes, recordings, bookmarks, and streaks. If the previous user was the review account (which bypasses paywalls), the paywall bypass effectively carries over to all subsequent users on that device.

**Reproduction:**
1. Log in with any account that has data (e.g., the review account with completed surahs)
2. Log out
3. Create/login with a completely new account
4. Observe: khatma shows completed surahs, insights show data, no paywall triggers

---

## 2. Root Cause

In `src/features/auth/infrastructure/AuthContext.tsx`:

- A `clearUserData(uid)` function exists (line 44-75) that removes all user-scoped AsyncStorage keys
- **BUT it's only called in `deleteAccount()` and `deleteAccountWithPassword()`** (lines 200 and 216)
- **The `logout()` function (line 175) does NOT call `clearUserData()`** — it only clears `hasSeenWelcome` and reading positions
- Additionally, the khatma key `khatma_progress_${year}` is **missing** from the `clearUserData` key list

---

## 3. The Fix

### File: `src/features/auth/infrastructure/AuthContext.tsx`

**Change 1:** Add the khatma storage key to `clearUserData()`:

```diff
 async function clearUserData(uid: string | undefined) {
     const keysToRemove = [
         // User-scoped mood data
         `mood_history_${uid}`,
         `mood_free_uses_${uid}`,
         `mood_today_${uid}`,
         // Streaks
         `reflection_streaks_${uid}`,
         'reflection_streaks',
         // User content
         'recordings',
         'folders',
         'user_notes',
         'bookmarks',
         // Reading data
         'reading_position',
         'reading_positions',
         'reading_activity_log',
         // Khatma
         'reading_log_v3_seeded',
         'khatma_celebration_shown_round',
         'khatma_onboarding_dismissed',
+        // Khatma progress (year-based key)
+        `khatma_progress_${new Date().getFullYear()}`,
         // Onboarding
         `@quran_notes:onboarding:${uid}`,
     ];
```

**Change 2:** Call `clearUserData()` in the `logout()` function:

```diff
 const logout = async () => {
     setLoading(true);

         setUser(null);

         // Clear welcome state for next user (Device Global)
         await AsyncStorage.removeItem('hasSeenWelcome');
     } catch (e) {
```

That's it. Two changes in one file.

## 4. Important Context

| Topic | Detail |
|---|---|
| **Review account** | `mahmoudahmedalaa+review@gmail.com` — hardcoded in `src/features/auth/infrastructure/ProContext.tsx` (line 6) to always get Pro access. This is correct and should stay. |
| **Paywall logic** | `ProContext.tsx` checks RevenueCat for `pro_access` entitlement, OR grants Pro if email matches the review account. The paywall itself is fine — the problem is stale local data, not the paywall code. |
| **Cloud data** | User data in Firestore is **not affected**. Notes, recordings are stored in Firestore. Clearing AsyncStorage doesn't touch cloud data. BUT some features are local-only (khatma, reading log, mood, streaks) — these will reset on logout. This is correct behavior since there's no cloud sync for these yet. |
| **Storage keys** | Khatma uses `khatma_progress_${year}`, mood uses `mood_history_${uid}` etc., streaks use `reflection_streaks_${uid}`, reading log uses `reading_activity_log`. All are in AsyncStorage. |
