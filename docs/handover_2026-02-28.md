## 2. Root Cause Analysis

### The Core Problem: Two Audio Systems Don't Talk to Each Other

The app has **two separate audio systems** that don't coordinate:

| System | File | What it manages |
|--------|------|-----------------|
| **AudioContext** | `src/features/audio-player/infrastructure/AudioContext.tsx` | TrackPlayer playback, `playingVerse` state, `isPlaying` state, `stop()` function |
| **useAudioRecorder** | `src/core/hooks/useAudioRecorder.ts` | expo-av Recording, `isRecording` state, `startRecording()` / `stopRecording()` |

### Bug 1: Why "recorder not prepared" happens

**Flow that crashes**:
1. User plays audio (TrackPlayer is active, `AudioContext.playingVerse` is set)
2. User taps record on a verse
3. `handleRecordVerse()` in `app/surah/[id].tsx:450` calls `startRecording()`
4. `startRecording()` in `useAudioRecorder.ts:54` calls `releaseAudioSession()` (line 60)
5. `releaseAudioSession()` calls `TrackPlayer.reset()` (line 47) — this stops audio at the native level
6. But `TrackPlayer.reset()` is async and iOS needs time to release the audio session
7. Only **100ms delay** (line 80) before attempting `Audio.Recording.createAsync()`
8. **iOS hasn't finished releasing the playback audio session** → `Audio.Recording.createAsync()` fails with "Prepare encountered an error: recorder not prepared"

**The fix**: `handleRecordVerse` must call `AudioContext.stop()` (which properly clears all state AND calls `TrackPlayer.stop()`) BEFORE calling `startRecording()`. Then increase the delay to 300-500ms. The `stop()` function is already available from `useAudio()` at line 50.

### Bug 2: Why the UI shows wrong state after the error

**Root cause**: `useAudioRecorder.releaseAudioSession()` calls `TrackPlayer.reset()` directly, bypassing `AudioContext`. So:
- `TrackPlayer.reset()` stops the native audio — audio stops streaming
- But `AudioContext.playingVerse` is still set (was never cleared)
- `AudioContext.isPlaying` may still be true (depends on event timing)
- The verse item checks `playingVerse` to show highlight + pause icon
- The sticky player checks `isPlaying` to show play/pause — gets the stale event

**The fix**: NEVER call `TrackPlayer.reset()` directly from `useAudioRecorder`. Instead, the surah screen should call `AudioContext.stop()` before starting recording, because `stop()` properly clears `playingVerse`, `isPlaying`, `playlist`, etc. (see `AudioContext.tsx:296-309`).

### Bug 3: Why recording fails permanently after the first error  

After the error in `startRecording()`:
1. The catch block (line 90-99) calls `forceCleanup()` — which sets `isRecording=false`, `isPaused=false`
2. But `Audio.setAudioModeAsync({ allowsRecordingIOS: true })` was already called (line 74)
3. The audio mode is stuck in recording mode even though no recording is active
4. Next `startRecording()` call: `releaseAudioSession()` tries `TrackPlayer.getPlaybackState()` but TrackPlayer was already reset — may throw or return unexpected state
5. `Audio.Recording.createAsync()` fails again because the audio session is in a bad state

**The fix**: In the catch block of `startRecording()`, after `forceCleanup()`, also call `Audio.setAudioModeAsync({ allowsRecordingIOS: false, playsInSilentModeIOS: true })` to properly restore the audio mode.

## 3. Files to Change

### File 1: `app/surah/[id].tsx`

**What to change**: `handleRecordVerse` (line ~450) and `handleRecordSurah` (line ~518)

Before calling `startRecording()`, both functions must:
1. Check if audio is currently playing (`isPlaying` from `useAudio()`)
2. If yes, call `await stop()` from AudioContext to properly stop playback and clear all state
3. Add a delay (300ms) after `stop()` to let iOS release the audio session
4. Then call `startRecording()`

```typescript
// Example fix for handleRecordVerse (line ~488):
const handleRecordVerse = async (verseId: number) => {
    // ... existing conflict handling for recording-on-different-verse ...
    
    // If already recording on same verse, ignore
    if (isRecording || isPaused) return;
    
    // CRITICAL: Stop any active playback BEFORE recording
    if (isPlaying) {
        await stop(); // AudioContext.stop() — clears playingVerse, isPlaying, etc.
        await new Promise(resolve => setTimeout(resolve, 300)); // Let iOS release session
    }
    
    setRecordingVerseId(verseId);
    await startRecording();
};
```

Same pattern for `handleRecordSurah`, and for the conflict handler `onPress` callbacks (Discard & Record, Save & Record).

### File 2: `src/core/hooks/useAudioRecorder.ts`

**Change 1**: Remove `releaseAudioSession()` from `startRecording()` OR make it a no-op if the caller already stopped playback. The problem is that `releaseAudioSession` calls `TrackPlayer.reset()` which bypasses AudioContext state management.

The cleanest approach: keep `releaseAudioSession()` as a safety net but understand that the PRIMARY stop should happen in the surah screen via `AudioContext.stop()`.

**Change 2**: In the catch block of `startRecording()` (line 90-99), add audio mode reset:

```typescript
catch (error) {
    if (__DEV__) console.error('Failed to start recording:', error);
    await forceCleanup();
    // CRITICAL: Reset audio mode so future recordings can work
    try {
        await Audio.setAudioModeAsync({
            allowsRecordingIOS: false,
            playsInSilentModeIOS: true,
        });
    } catch { /* ignore */ }
    const errorMsg = error instanceof Error ? error.message : 'Unknown error';
    Alert.alert(
        'Recording Error',
        `Could not start recording: ${errorMsg}. Make sure no other audio is playing and try again.`,
    );
}
```

**Change 3**: Increase the delay from 100ms to 500ms (line 80):
```typescript
await new Promise(resolve => setTimeout(resolve, 500));
```

### File 3: `src/features/audio-player/infrastructure/AudioContext.tsx`

No changes needed in this file. Its `stop()` function (line 296-309) already properly clears everything. The bug is that it was never being called before recording.

## 4. Critical Architecture Context

### AudioContext.stop() (line 296-309) does ALL of these:
```
setIsPlaying(false)
setPlayingVerse(null)         ← clears verse highlight
await player.stop()           ← stops TrackPlayer
setPlaylist([])               ← clears queue
setCurrentSurahNum(null)      ← clears surah tracking
setCurrentSurahName(null)
```

### useAudioRecorder.releaseAudioSession() (line 42-52) only does:
```
await TrackPlayer.reset()     ← stops native audio ONLY, no state cleanup
```

**This mismatch is the root of ALL three bugs.**

### Key import in surah/[id].tsx (line 50):
```typescript
const { playingVerse, isPlaying, isRecordingActive, playFromVerse, pause, resume, stop, setRecordingActive } = useAudio();
```
`stop` is already imported and available. It just needs to be called.
