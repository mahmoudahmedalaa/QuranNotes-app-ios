## 4. Critical Context

**Architectural decisions:**
- Recording cleanup uses `forceCleanup()` before every `startRecording()` — this is the key to preventing the "Only one Recording" crash. The previous approach relied on `stopRecording()` cleanup which failed when navigation interrupted the flow.
- Donut values are now **raw minutes** (not percentages). The PieChart library sizes slices proportionally to values. A 2% minimum floor ensures tiny categories are always visible.
- AppState listener is inside `useAudioRecorder` (not the screen) so it works regardless of which screen hosts recording.

**Tricky bugs solved:**
- **"Only one Recording" crash**: expo-av's `Audio.Recording` is a singleton — calling `prepareToRecordAsync()` twice without unloading the first throws a fatal. `forceCleanup()` always stops + unloads before starting fresh.
- **Donut slices disappearing**: `Math.round((2/670)*100) = 0` — tiny categories rounded to zero. Fixed by using raw minutes as values with a minimum floor.
- **Grey Arabic text**: Color was correctly `#000000` but `opacity: 0.4` on completed dhikrs made it look grey. The "Done" checkmark already signals completion; opacity dimming was redundant and harmful to readability.
- **Donut slices turning white on tap**: `showGradient` was trying to apply a gradient URL that didn't exist. Removed entirely; `extraRadius` achieves the same pop-out effect correctly.

**User preferences:**
- Avoid big code rewrites — targeted, minimal changes preferred
- No hardcoded values without reason
- Arabic text must always be clearly readable (black on light background)
- All recordings (even < 1 minute) should count toward the donut via `Math.ceil`
- User is preparing for App Store submission — stability is critical
