## 4. Critical Context

### 4.1 The Audio Bug Root Cause (MUST UNDERSTAND)

**File**: `src/features/audio-player/infrastructure/AudioContext.tsx`

The app has a playback event listener (line ~207) that receives ALL status events from `react-native-track-player`. A previous session added `reciterSwitchingRef` to prevent a "verse 1 flash" when switching reciters. The bug was:

```tsx
// BEFORE (BROKEN) ŌĆö blocked ALL events including isPlaying, didJustFinish
if (reciterSwitchingRef.current) {
    return; // ŌåÉ THIS WAS THE BUG. Full early return = UI freezes
}
```

```tsx
// AFTER (FIXED) ŌĆö only blocks verse detection, events still pass through
setIsPlaying(status.isPlaying); // ŌåÉ Always updates play state
if (!reciterSwitchingRef.current) {
    // verse detection code here (guarded)
}
// didJustFinish handler below (unguarded, always runs)
```

**Why this matters**: When switching reciters, it takes 1-5 seconds to fetch audio from Quran.com API and load tracks. During that time, the old guard blocked ALL events. The UI showed play button as unresponsive because `setIsPlaying(status.isPlaying)` was never called. The `didJustFinish` handler was also blocked, meaning queues could end without the app knowing.

### 4.2 The `reciterSwitchingRef` Lifecycle

1. Set to `true` at reciter switch (line ~159)
2. 10-second safety timeout starts (line ~161) ŌĆö guaranteed to clear it
3. `startFullSurahPlayback()` is called ŌåÆ network fetch ŌåÆ load tracks
4. `.finally()` handler clears it after 500ms delay (line ~184/199)
5. Worst case: 10s safety timeout clears it

**Risk**: If the user switches reciters rapidly (e.g., 3 times in 2 seconds), each switch sets the flag to `true`. The `.finally()` from the first switch could clear the flag while the third switch is still loading. The safety timeout handles this worst case.

### 4.3 Dual Playback Modes

The app has TWO playback architectures:
- **Full-surah mode** (preferred): Streams a single MP3 of the entire surah, detects verses via timestamp polling. Uses Quran.com API (`quranComId`).
- **Per-verse mode** (fallback): Loads individual verse MP3s into a queue with pre-download caching (4-second timeout per verse).

The new guard `!player.getIsFullSurahMode()` on line ~238 prevents per-verse track change events from setting `playingVerse` when in full-surah mode. Without this guard, switching to full-surah mode could briefly trigger verse detection from the old per-verse queue (the "verse 1 flash").
